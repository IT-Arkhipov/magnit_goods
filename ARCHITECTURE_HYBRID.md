# Архитектура Product Scanning Workflow (Обновлено)

**Дата обновления:** 2026-04-14  
**Статус:** Архитектура переработана для гибридного подхода

---

## Новая архитектура

### Источники данных категорий

**JSON файл (`magnit_catalog.json`):**
- Содержит **только корневые категории** (18 шт)
- Загружается один раз при инициализации
- Используется как базовая структура каталога
- Не требует обновления часто

**API Magnit (`/webgate/v1/goods/filters`):**
- Получает **подкатегории** динамически
- Вызывается при нажатии "Обновить каталог"
- Обеспечивает актуальность подкатегорий
- Поддерживает полную синхронизацию (добавление, обновление, удаление)

---

## Workflow

### 1. Инициализация (один раз)

```
POST /api/categories/load-from-json
↓
load_catalog_from_json() читает magnit_catalog.json
↓
Вставляет 18 корневых категорий в БД
↓
Подкатегории НЕ загружаются из JSON
↓
Результат: {"scanned": 18, "added": 18, "updated": 0}
```

**Что происходит:**
- Загружаются только корневые категории (parent_id=None)
- Подкатегории из JSON игнорируются
- Каждая корневая категория имеет magnit_id для связи с API

---

### 2. Обновление каталога (вручную по кнопке)

```
POST /api/categories/update-catalog?store_code=992104
↓
CatalogScanner.scan_categories() вызывает API
↓
API возвращает ВСЕ категории (корневые + подкатегории)
↓
Для каждой категории:
  - Если новая: добавить
  - Если существует: обновить (name, url, product_count)
  - Если нет в API: удалить (полная синхронизация)
↓
Результат: {"scanned": N, "added": X, "updated": Y, "deleted": Z}
```

**Что происходит:**
- Получаются все категории из API
- Корневые категории обновляются (если изменились)
- Новые подкатегории добавляются
- Устаревшие подкатегории удаляются
- Корневые категории из JSON НЕ удаляются (защита)

---

### 3. Сканирование товаров

```
POST /api/catalog/scan-all-stores
↓
Для каждого магазина:
  CatalogScanner.scan_products(
    category_ids=[cat.magnit_id for tracked cats],
    tracked_only=False
  )
↓
Для каждой категории:
  MagnitAPIClient.get_products(
    category_ids=[magnit_id],
    store_code=code,
    limit=50, offset=0
  )
↓
Товары сохраняются в БД с price history и daily snapshots
```

**Что происходит:**
- Используются категории, которые уже в БД (загруженные из JSON + API)
- Сканируются только отслеживаемые категории (is_tracked=True)
- Товары связываются с категориями по magnit_id

## API Endpoints

### Инициализация

**POST /api/categories/load-from-json**
- Загружает корневые категории из JSON
- Вызывается один раз при первой инициализации
- Ответ: `{"status": "completed", "scanned": 18, "added": 18, "updated": 0}`

### Обновление каталога

**POST /api/categories/update-catalog?store_code=992104**
- Сканирует подкатегории из API
- Вызывается вручную при нажатии "Обновить каталог"
- Полная синхронизация (добавление, обновление, удаление)
- Ответ: `{"status": "completed", "scanned": N, "added": X, "updated": Y, "deleted": Z}`

### Получение дерева категорий

**GET /api/categories/tree**
- Возвращает иерархическое дерево всех категорий
- Включает корневые категории и подкатегории
- Ответ: Массив корневых категорий с вложенными children

### Выбор категорий для отслеживания

**POST /api/categories/update-tracking**
- Устанавливает is_tracked=True для выбранных категорий
- Остальные категории получают is_tracked=False
- Ответ: `{"status": "success", "tracked_count": N, "updated_ids": [...]}`

### Сканирование товаров

**POST /api/catalog/scan-all-stores**
- Сканирует товары для всех магазинов
- Использует отслеживаемые категории
- Асинхронное выполнение с отслеживанием прогресса
- Ответ: `{"job_id": N, "status": "pending"}`

## Преимущества гибридного подхода

✅ **JSON для корневых категорий:**
- Стабильная структура
- Не требует частого обновления
- Быстрая инициализация
- Контролируемая иерархия

✅ **API для подкатегорий:**
- Всегда актуальные данные
- Автоматическое обнаружение новых подкатегорий
- Удаление устаревших подкатегорий
- Гибкость при изменениях в каталоге Magnit

✅ **Полная синхронизация:**
- Гарантирует консистентность данных
- Удаляет "мусор" из БД
- Защищает корневые категории от удаления

## Сценарии использования

### Сценарий 1: Первая инициализация

```bash
# 1. Загружаем корневые категории из JSON
POST /api/categories/load-from-json
# Результат: 18 корневых категорий в БД

# 2. Обновляем каталог - получаем подкатегории из API
POST /api/categories/update-catalog?store_code=992104
# Результат: 18 корневых + N подкатегорий в БД

# 3. Выбираем категории для отслеживания
POST /api/categories/update-tracking
# Body: {"category_ids": [1, 2, 3, ...]}

# 4. Сканируем товары
POST /api/catalog/scan-all-stores
# Результат: товары сохранены в БД
```

### Сценарий 2: Обновление каталога

```bash
# Magnit добавил новые подкатегории или изменил существующие
# Нажимаем кнопку "Обновить каталог"

POST /api/categories/update-catalog?store_code=992104
# Результат:
# - Новые подкатегории добавлены
# - Существующие обновлены
# - Удалённые подкатегории удалены из БД
```

### Сценарий 3: Переинициализация

```bash
# Нужно полностью переинициализировать каталог

# 1. Удаляем все категории из БД (вручную или через API)
DELETE FROM categories

# 2. Загружаем корневые категории из JSON
POST /api/categories/load-from-json

# 3. Обновляем каталог
POST /api/categories/update-catalog?store_code=992104
```

## Обработка ошибок

### Если API недоступен при обновлении каталога

- Retry логика: 3 попытки с exponential backoff (2, 4, 8 секунд)
- Если все попытки исчерпаны: возвращается ошибка 500
- Корневые категории остаются в БД (не удаляются)

### Если подкатегория удалена в API

- При следующем обновлении каталога подкатегория удаляется из БД
- Товары, связанные с подкатегорией, остаются в БД (orphaned)
- Рекомендуется: удалить orphaned товары или переместить их

## Миграция с старой архитектуры

Если раньше использовалась полная загрузка из JSON:

1. Сохраните текущие данные (backup БД)
2. Обновите `load_catalog_from_json.py` (новая версия)
3. Очистите таблицу categories
4. Запустите новую инициализацию:
   - `POST /api/categories/load-from-json` (корневые категории)
   - `POST /api/categories/update-catalog` (подкатегории из API)
5. Переустановите отслеживание категорий
