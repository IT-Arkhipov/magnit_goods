# Статусы заданий на сканирование (scan_jobs)

## Зачем нужны

Статусы заданий на сканирование (`scan_jobs`) нужны по нескольким причинам:

### 1. Сканирование — длительная операция

Полное сканирование 5000 товаров может занять **10-30 минут** (пагинация, задержки между запросами). Клиент не может ждать HTTP-ответ столько времени.

**Решение:** сервер сразу возвращает `job_id`, а сканирование идёт в фоне. Клиент периодически спрашивает: «Какой статус job #123?»

```
POST /api/catalog/scan  →  {"job_id": 42, "status": "pending"}
GET /api/jobs/42        →  {"status": "running", "items_scanned": 1200}
GET /api/jobs/42        →  {"status": "running", "items_scanned": 3800}
GET /api/jobs/42        →  {"status": "completed", "items_added": 5000}
```

### 2. Отслеживание прогресса

Поля `items_scanned`, `items_added`, `items_updated` показывают, что процесс идёт и не завис:

```json
{
  "status": "running",
  "items_scanned": 2300,
  "items_added": 450,
  "items_updated": 1850
}
```

UI может рисовать прогресс-бар.

### 3. Обработка ошибок

Если сканирование упало (сайт недоступен, API изменился), статус `failed` + `error_message`:

```json
{
  "status": "failed",
  "error_message": "Connection timeout after 3 retries",
  "items_scanned": 800
}
```

Пользователь видит конкретную ошибку, а не «что-то пошло не так».

### 4. История запусков

Можно посмотреть, когда было последнее сканирование, сколько длилось, сколько товаров добавлено:

```sql
SELECT * FROM scan_jobs
WHERE job_type = 'prices'
ORDER BY created_at DESC
LIMIT 10;
```

### 5. Предотвращение параллельных запусков

Перед запуском нового сканирования проверяем — нет ли уже running-задания того же типа:

```python
running = db.query(ScanJob).filter(
    ScanJob.job_type == 'prices',
    ScanJob.status == 'running'
).first()
if running:
    raise HTTPException("Сканирование уже запущено")
```

## Альтернативы и почему не подходят

| Подход | Почему не подходит |
|--------|-------------------|
| Синхронный запрос | Браузер timeout через 30-60 сек, пользователь не знает, что происходит |
| WebSocket | Избыточно для 1 пользователя, сложнее реализация |
| Просто лог-файл | Нет структуры, нельзя показать прогресс в UI |
| APScheduler без трекинга | Непонятно, выполнилось ли задание, были ли ошибки |

**Вывод:** `scan_jobs` — это минимальный и достаточный механизм для управления долгими фоновыми операциями в однопользовательском приложении.
